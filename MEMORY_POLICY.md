# Memory Policy

## Store

Persist:

- stable preferences
- operating style
- founder priorities
- recurring workflows
- project-level decisions
- active bets and themes
- important constraints
- approved tool and autonomy boundaries

## Do Not Store

- raw secrets
- access tokens
- transient chatter
- low-signal conversation filler
- redundant copies of documents that already exist in git

## Compression Rule

- Compress aggressively.
- Keep only load-bearing facts.
- Prefer durable summaries over verbose transcripts.

## Update Rule

- Update memory when a preference or decision is stable enough to matter again.
- Do not update memory for one-off noise.

## Retrieval Rule

- Retrieve only the minimum needed context.
- Stable preferences beat noisy full-history recall.

## Git Over Memory

- If something is a durable operating rule, spec, or decision, prefer storing it in git.
- Memory supplements the repo. It does not replace versioned source-of-truth files.
